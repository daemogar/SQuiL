SQuiL Writing Guide

SQuiL turns annotated .squil files into strongly-typed C# code via Roslyn source generation. You write SQL that looks like SQL — SQuiL handles the plumbing. (.squil is the canonical extension; plain .sql files are also accepted.)

Commands: open the Command Palette (F1 / Ctrl+Shift+P) and type SQuiL to reach New File, Preview Generated C#, Build Project, Open Writing Guide, and Check for Updates. The extension installs from the VS Code Marketplace (search “SQuiL”).
Commands: the SQuiL commands live on Tools ▸ SQuiL — Preview Generated C#, Build SQuiL Project, SQuiL Writing Guide, and Check for Updates; New SQuiL File is on File ▸ New. SSMS's SQL-editor right-click menu can't host third-party commands, so use the Tools menu. Because .squil opens in SSMS's SQL Query Editor, you also get full F5 execution and a live results grid against any connection. Install with the bundled install.cmd.

File Structure

Every SQuiL file follows this three-part layout:

--Name: QueryName          ← optional, overrides filename as the C# class name

-- ① Header: DECLARE all variables (special variables like @Debug go first)
Declare @Debug           Bit = 1;
Declare @Param_UserID   Int;
Declare @Return_Username VarChar(100);

-- ② Use statement: must appear once, after all DECLAREs
Use [MyDatabase];

-- ③ Body: your SQL logic
Set @Return_Username = (Select Name From Users Where ID = @Param_UserID);
Select @Return_Username;
IntelliSense tips:
  • In the header, typing @ on a blank line auto-prepends Declare.
  • In the body (after Use), typing @ shows only the variables already declared above the cursor.
  • Editing the middle of an existing @… token — e.g. changing @Param_X to @Params_X — is a plain text edit: completions stay out of your way.

Variable Naming Conventions

The prefix determines the variable's role and what C# code gets generated.

PatternRoleGenerated C#
@Param_Name Input scalar Property on *Request record
@Params_Name Input table-valued IEnumerable<NameTable> on *Request
@Param_Name Table(...) Input object NameObject record on *Request
@Return_Name Output scalar Property on *Response record
@Returns_Name Output table IEnumerable<NameTable> on *Response
@Return_Name Table(...) Output object NameObject record on *Response
@Debug Special Opt-in. On *Request as bool Debug only when you declare it in SQL — nothing is emitted implicitly. Declare it first in the header (see Variable Rules below).
@SuppressDebug Special Opt-in. On *Request as bool SuppressDebug when declared. Requires @Debug to also be declared — declaring it alone is a build error (diagnostic SP0019). It gates the auto-debug expression: SQuiL sends !request.SuppressDebug && (request.Debug || EnvironmentName != "Production"). (Replaces the old DebugOnly property, which no longer exists.)
@AsOfDate Special Opt-in. A caller-supplied point-in-time, emitted on *Request as a nullable typed property (the type follows the SQL type map — e.g. date → System.DateOnly?). When the caller leaves it null, the current time is substituted at execution; any SQL initializer (e.g. = '2008-10-01') is ignored at runtime. Distinguished from an ordinary @Param_AsOfDate by being declared bare (no @Param_ prefix).
@EnvironmentName Special Opt-in. Sent as a SQL parameter only when declared (not emitted as a *Request property). Resolved by SQuiLBaseDataContext from IConfiguration (key "EnvironmentName") or the environment variable. Declare in SQL only when the query body needs to read it.

Variable Rules

A SQuiL file is valid T-SQL, full stop. SQuiL never renames or remaps a variable for you, so the same rule SQL Server enforces at batch compile time applies in the editor and at build time:

  • Declare before use. Every @variable you reference must have a Declare above the first reference — referencing @PersonID when you declared @Param_PersonID is an error, not a shorthand. The build fails with diagnostic SP0013, and the editor marks the reference.
  • Special variables are opt-in, and not exempt from declare-before-use. The four input specials — @Debug, @SuppressDebug, @EnvironmentName, and @AsOfDate — only affect the generated C# when the file declares them, and the SQL body may only reference them if declared. @SuppressDebug additionally requires @Debug to be declared; declaring it alone fails the build with diagnostic SP0019.
  • Declare the input specials in the header, first. All four input specials (@Debug, @SuppressDebug, @EnvironmentName, @AsOfDate) share the same placement rules: declaring one after the Use statement is an error (diagnostic SP0016); declaring one after other header declarations produces a warning — put them at the very top.
  • Table variables that share a name share one record — and must share one shape. Declaring @Param_User Table(...) in one place and @Return_User Table(...) in another (in the same file or across query files) generates a single UserObject/UserTable record, so both declarations must list identical columns — same names, same types, same nullability, same order (sizes like varchar(50) vs varchar(100) may differ). A mismatch fails the build with diagnostic SP0017; either align the column lists or rename one of the variables.
-- ✔ valid: declared first, referenced by exact name
Declare @Debug          bit = 0;
Declare @Param_PersonID varchar(10);

Use [MyDatabase];

Select * From People Where PersonID = @Param_PersonID;

-- ✘ error SP0013: `@PersonID` was never declared (no prefix shorthand)
--   Select * From People Where PersonID = @PersonID;

Variables declared in the body (after Use) are ordinary T-SQL locals — they need no SQuiL prefix, but they still must be declared before they are referenced.

Examples

1. Lookup by ID — scalar in, scalar out

--Name: GetUserName

Declare @Param_UserID int;
Declare @Return_Name  varchar(100);

Use HrDatabase;

Set @Return_Name = (
    Select u.Name
    From   Users u
    Where  u.UserID = @Param_UserID
);

Select @Return_Name;

Generated C#:

public partial record GetUserNameRequest
{
    public int UserID { get; set; }
}

public partial record GetUserNameResponse
{
    public string Name { get; set; }
}

// Method emitted into your UserDataContext partial class:
public async Task<SQuiLResultType<GetUserNameResponse>> ProcessGetUserNameAsync(
    GetUserNameRequest request,
    CancellationToken cancellationToken = default!) { /* generated */ }

2. Filtered list with optional include-flag — table-valued output

--Name: ListUsersByDepartment

Declare @Param_DepartmentID    int;
Declare @Param_IncludeInactive bit = 0;
Declare @Returns_Users         table (
    UserID   int,
    Name     varchar(100),
    Email    varchar(255),
    HireDate date
);

Use HrDatabase;

Insert Into @Returns_Users
Select      u.UserID, u.Name, u.Email, u.HireDate
From        Users u
Where       u.DepartmentID = @Param_DepartmentID
            And (@Param_IncludeInactive = 1 Or u.IsActive = 1);

Select * From @Returns_Users;

Generated C#:

public partial record ListUsersByDepartmentRequest
{
    public int DepartmentID { get; set; }
    public bool IncludeInactive { get; set; }
}

public partial record UsersTable
{
    public int UserID { get; set; }
    public string? Name { get; set; }
    public string? Email { get; set; }
    public DateOnly HireDate { get; set; }
}

public partial record ListUsersByDepartmentResponse
{
    public List<UsersTable>? Users { get; set; } = [];
}

// Method emitted into your UserDataContext partial class:
public async Task<SQuiLResultType<ListUsersByDepartmentResponse>> ProcessListUsersByDepartmentAsync(
    ListUsersByDepartmentRequest request,
    CancellationToken cancellationToken = default!) { /* generated */ }

3. Bulk write — table-valued input with row count out

--Name: AssignTagsToArticle

Declare @Param_ArticleID    int;
Declare @Params_Tags        table (
    Label  nvarchar(100),
    Weight int
);
Declare @Return_RowsInserted int;

Use ContentDatabase;

Insert Into ArticleTags (ArticleID, Label, Weight)
Select      @Param_ArticleID, t.Label, t.Weight
From        @Params_Tags t;

Set @Return_RowsInserted = @@RowCount;
Select @Return_RowsInserted;

Generated C#:

public partial record TagsTable
{
    public string? Label { get; set; }
    public int Weight { get; set; }
}

public partial record AssignTagsToArticleRequest
{
    public int ArticleID { get; set; }
    public List<TagsTable>? Tags { get; set; } = [];
}

public partial record AssignTagsToArticleResponse
{
    public int RowsInserted { get; set; }
}

// Method emitted into your ContentDataContext partial class:
public async Task<SQuiLResultType<AssignTagsToArticleResponse>> ProcessAssignTagsToArticleAsync(
    AssignTagsToArticleRequest request,
    CancellationToken cancellationToken = default!) { /* generated */ }

4. Using @Debug for diagnostic output

@Debug is opt-in: declare it when you want a bool Debug on the request and the SQL body to be able to read the value. Put the declaration first in the header. If you also declare @SuppressDebug (which requires @Debug), a bool SuppressDebug appears on the request too and gates the auto-debug expression:

--Name: ListOrdersByStatus

Declare @Debug          bit = 1;
Declare @Param_Status   varchar(20);
Declare @Returns_Orders table (
    OrderID int,
    Total   decimal(18, 2),
    Placed  datetime
);

Use OrdersDatabase;

If @Debug = 1
    Print Concat('[ListOrdersByStatus] status = ', IsNull(@Param_Status, '(any)'));

Insert Into @Returns_Orders
Select      o.OrderID, o.Total, o.PlacedOn
From        Orders o
Where       @Param_Status Is Null Or o.Status = @Param_Status;

Select * From @Returns_Orders;
Tip: the @Debug default of = 1 is used only when you execute the file directly against a connection (F5) — its results land in the SSMS grid. At runtime SQuiL overrides it with request.Debug, so C# callers stay in control. Prefer Concat() over string + in diagnostics — it tolerates Null instead of producing one.

Sample Data Insertion

For local testing, you can have SQuiL generate an Insert Into block populated with type-aware sample rows for any @Params_ (list) or @Param_… Table(...) (single object) variable.

On the line immediately after the variable's Declare, type @. The completion list will include a ⊕ Insert sample data → @Params_Tags entry — that IntelliSense completion is the only way to insert sample data; there is no menu, palette, or right-click command for it. Choosing it:

  • For @Params_ (list): prompts for the number of rows, then inserts an Insert Into block with that many records.
  • For @Param_… Table(...) (single object): silently inserts exactly one row.

If a sample block already exists for that variable, the completion reads ⊕ Modify sample data → … and re-running it replaces the existing block in place. Existing blocks are detected by the Insert Into @Param_… statement itself — there are no comment markers, so a hand-written Insert Into for that variable is also detected and can be regenerated.

Declare @Params_Tags Table (Label NVarChar(100), Weight Int);
-- ↓ generated by the ⊕ Insert sample data completion
Insert Into @Params_Tags (Label, Weight)
Values
    ('Label 1', 1),
    ('Label 2', 2),
    ('Label 3', 3);
⚠ Sample data is intended for local testing only — remove it (or guard it behind @Debug) before committing.

Connection String Configuration

SQuiL uses the database name from the Use statement as the connection string key:

// appsettings.json
{
  "ConnectionStrings": {
    "HrDatabase": "Server=.;Database=HR;Trusted_Connection=True;"
  }
}

To use a different key, pass it to the [SQuiLQuery] attribute:

[SQuiLQuery(QueryFiles.GetUserCount, setting: "CustomKey")]
public partial class GetUserCountDataContext { }

Project Setup

1. NuGet reference

<PackageReference Include="SQuiL.SourceGenerator" Version="*" />

2. Register your .squil files as AdditionalFiles

SQuiL discovers queries via MSBuild's AdditionalFiles item. The simplest pattern picks up every .squil file anywhere in the project — the file's relative path becomes the entry in the generated QueryFiles enum:

<ItemGroup>
    <AdditionalFiles Include="**\*.squil" />
</ItemGroup>

You can scope it tighter if you'd rather keep queries in a dedicated folder:

<AdditionalFiles Include="Queries\**\*.squil" />
Plain .sql query files are also accepted — add a second AdditionalFiles glob for *.sql if your project mixes both. Backslash separators are fine cross-platform; MSBuild normalizes them (forward slashes work too).

3. Declare the data context ([SQuiLQuery])

Your data context is a partial class decorated with one [SQuiLQuery] attribute per .squil file it owns. Inheriting SQuiLBaseDataContext explicitly is not required — SQuiL supplies the base type and an IConfiguration constructor automatically when the class declares no constructor of its own. Declaring any constructor (primary or ordinary) opts out; that constructor must then chain : base(configuration).

using SQuiL;

namespace YourApp.Data;

[SQuiLQuery(QueryFiles.GetUserName)]
[SQuiLQuery(QueryFiles.ListUsersByDepartment, setting: "HrDatabase")]
public partial class UserDataContext { }

Two things to note:

  • One partial class can host multiple [SQuiLQuery] attributes — each one adds a Process<QueryName>Async method to that context.
  • QueryFiles is an enum the source generator emits from your AdditionalFiles; the member name comes from the SQL filename (or its --Name: override).

4. Register SQuiL for DI

SQuiL emits an AddSQuiL() extension method that registers every generated data context for you. Call it once in Program.cs:

builder.AddSQuiL();

That's the whole DI setup. You do not need to register IConfiguration (the host builder already provides it) or each data context individually (AddSQuiL handles them). The base class resolves connection strings from IConfiguration.GetConnectionString(setting); the optional setting: argument on [SQuiLQuery] selects the key, and if omitted SQuiL uses SQuiLDatabase as the default key.

5. Shared table types (optional)

If multiple queries return the same row shape, declare it once with [SQuiLTable]:

[SQuiLTable(TableType.User)]
public partial class Tables { }

SQuiL will then reuse the generated UserTable record across every query whose @Returns_User Table(...) declaration matches that shape.

Two rules apply to shared table types:

  • Every declaration that feeds one record must declare identical columns — same names, types, nullability, and order (sizes may differ). This covers table variables that share a name and different names mapped to one class via multiple [SQuiLTable] attributes. A mismatch fails the build with diagnostic SP0017.
  • A [SQuiLTable] partial record must not declare a primary constructor — the generator emits the parameter list, and C# allows only one partial declaration to have one. Use a body instead; diagnostic SP0018 flags violations.
// ✔ customize with a body — add members freely
[SQuiLTable(TableType.User)]
public partial record UserTable { }

// ✘ error SP0018: the generator owns the parameter list
// [SQuiLTable(TableType.User)]
// public partial record UserTable(string UserID);

Solution Explorer File Nesting (.filenesting.json)

Drop a .filenesting.json at your solution or project root so Visual Studio tucks each .squil file under its companion .cs partial-class file in Solution Explorer (and gives Razor / appsettings the same treatment):

{
    "help": "https://go.microsoft.com/fwlink/?linkid=866610",
    "dependentFileProviders": {
        "add": {
            "extensionToExtension": {
                "add": {
                    ".squil": [ ".cs" ],
                    ".sql": [ ".cs" ]
                }
            }
        }
    }
}
This is cosmetic only — it does not affect compilation or source generation. The matching rule is by base filename: GetUserName.squil nests under GetUserName.cs. Pair each query file with a same-named C# partial-class shard to take advantage of it.
